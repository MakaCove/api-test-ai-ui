# API 自动化测试 AI 平台 — 完整需求文档

本文档基于项目实现与产品讨论整理，作为功能边界、验收与后续迭代的参考。

---

## 一、项目背景与目标

### 1.1 背景

- 接口测试依赖大量手工编写用例与自动化代码，效率低、易遗漏。
- 接口文档形态多样（Word、Markdown、纯文本），人工解析成本高。
- 希望借助 AI 能力，从文档或结构化接口信息自动生成测试用例与可执行的 JUnit 5 测试代码。

### 1.2 目标

- **多项目管理**：支持创建多个项目，每个项目独立管理文档、接口、用例与测试代码。
- **文档驱动**：支持上传/粘贴接口文档（Word、Markdown、文本），先由 AI 标准化，再提取为结构化接口信息。
- **手工补充**：支持不依赖文档，直接手工录入接口信息。
- **AI 生成**：根据接口信息生成测试用例；根据用例生成 JUnit 5 风格的 Java 自动化测试代码。
- **统一 Web 界面**：登录后在一个平台内完成文档管理、接口维护、用例管理、代码生成与导出，无需切换多套系统。
- **单体、前后端不分离**：采用 Spring Boot + Thymeleaf 服务端渲染 + 前端调用 REST API 的方式，便于部署与维护。

### 1.3 范围

- **在范围内**：用户注册/登录、项目管理、文档管理、接口管理、测试用例管理、测试代码生成与导出、系统配置（AI 与代码输出路径）。
- **暂不在范围内**：多环境（dev/test/prod）维度、细粒度权限（RBAC）、CI 集成、接口实际执行与报告、Swagger/OpenAPI 直接导入（预留扩展）。

## 二、术语与缩写

| 术语 | 说明 |
|------|------|
| 项目 | 被测系统或业务域的集合单元，下挂文档、接口、用例、测试代码 |
| 文档 | 接口说明资料，支持文件上传或文本粘贴，可被 AI 标准化并解析为接口 |
| 标准化内容 | 由 AI 对原始文档整理后的结构化文本，作为「AI 提取接口信息」的输入 |
| 接口 / API | 结构化接口定义：路径、方法、请求/响应结构等，来源为文档解析或手工录入 |
| 测试用例 | 针对单个接口的一条测试场景（请求数据、预期响应、校验规则、类型、优先级等） |
| 测试代码 | 根据用例生成的 JUnit 5 Java 测试类，可保存到本地工程或下载 |
| 软删除 | 通过标记字段（如 deleted_at）逻辑删除，列表与详情查询时过滤，不物理删库 |
| AI 提取接口信息 | 根据文档的标准化内容，调用 AI 解析出接口列表并写入库 |
| AI 生成用例 | 根据接口信息调用 AI 生成多条测试用例并写入库 |
| AI 生成测试用例自动化代码 | 根据单条或若干用例调用 AI 生成对应 JUnit 5 测试代码并写入库 |

## 三、用户角色

| 角色 | 说明 | 权限 |
|------|------|------|
| 未登录用户 | 访客 | 仅可访问登录页、注册页；访问业务页时跳转登录 |
| 已登录用户 | 注册并登录的用户 | 可使用全部功能：项目/文档/接口/用例/测试代码/系统配置；登出 |

当前版本不区分子角色（如管理员/普通用户），所有登录用户权限一致。角色字段（如 `role`）预留便于后续扩展。

## 五、非功能需求

### 5.1 性能与稳定性

- 文档标准化、AI 提取接口、AI 生成用例、AI 生成代码均为耗时操作，需合理超时与错误提示。
- AI 调用失败时给出明确提示，并记录日志；不向前端暴露敏感信息（如完整 Key）。
- 涉及写库的 AI 流程：AI 调用在事务外执行，仅在写库时开启短事务（先软删再插入），避免长事务导致锁等待超时。

### 5.2 安全

- 所有业务功能需登录后访问。
- 密码必须加密存储（BCrypt）。
- 敏感配置（数据库密码、AI Key）通过配置文件或环境变量管理，不提交到版本库。

### 5.3 可维护与可扩展

- Prompt 集中管理（如 AiPrompt），便于调优与版本化。
- AI 调用统一封装（如 AiClientService），便于更换模型或接口。
- 数据模型与接口设计预留对 Swagger/OpenAPI 等导入的扩展空间。

### 5.4 日志

- 关键操作（登录、注册、文档保存、AI 调用、用例/代码生成等）打日志。
- 日志输出到控制台与项目目录下 logs/application.log；按日滚动，保留一定天数。
- 日志级别可配置（如 root=INFO，org.example=DEBUG）。

### 5.5 数据与查询

- 所有列表类查询按创建时间倒序。
- 所有列表分页，每页 10 条；接口返回含 total、page、size、records。
- 日期时间统一格式：yyyy-MM-dd HH:mm:ss（接口返回与前端展示一致）。

### 5.6 UI/UX

- 全局统一布局：左侧侧栏导航，右侧主内容区；侧栏底部为「退出登录」。
- 主题保持深色系，点缀色（如青/绿渐变）统一；登录/注册、工作台、列表与详情风格一致。
- 列表分页样式统一，便于翻页与查看总数。
- 接口「修改」使用弹窗而非行内展开，避免页面过长。

## 六、领域模型与数据

### 6.1 实体关系概览

- **用户（User）**：登录账号，用户名唯一，密码加密。
- **项目（Project）**：一级管理单元；可软删（deleted）。
- **文档（Document）**：属于项目；含原始内容、标准化内容、状态等。
- **接口（ApiInfo）**：属于项目，可关联来源文档；含路径、方法、请求/响应结构、状态等。
- **测试用例（TestCase）**：属于项目与接口；含用例名、类型、请求数据、预期响应、校验规则、优先级、状态；支持软删（deleted_at）。
- **测试代码（TestCode）**：属于项目与接口，关联单条用例（test_case_id）；含语言、框架、类名、代码全文、状态；支持软删（deleted_at）。
- **系统配置（SystemSetting）**：键值对，存 AI 与代码输出等配置。

### 6.2 主要表结构（参考 schema.sql）

- t_user：用户表。
- t_project：项目表（deleted 逻辑删除）。
- t_document：文档表（project_id、original_content、standardized_content、status）。
- t_api_info：接口信息表（project_id、document_id、api_name、api_path、http_method、request_params、response_schema、status）。
- t_test_case：测试用例表（project_id、api_id、case_name、case_type、request_data、expected_response、validation_rules、priority、status、deleted_at）。
- t_test_code：测试代码表（project_id、api_id、test_case_id、language、framework、class_name、code_content、status、deleted_at）。
- t_system_setting：系统配置表（config_key、config_value）。

具体字段以 `src/main/resources/schema/schema.sql` 为准。

## 七、界面与导航结构

- **工作台**：`/`
- **项目列表**：`/projects` → 项目详情 → 文档管理 / 接口管理
- **文档管理**：`/projects/{id}/documents`
- **接口管理**：`/projects/{id}/apis`
- **某接口的用例列表**：`/projects/{pid}/apis/{apiId}/testcases`
- **某接口的测试代码列表**：`/projects/{pid}/apis/{apiId}/testcodes`
- **全局接口列表**：`/apis` → 详情 → 跳转至该接口的用例列表页
- **全局用例列表**：`/testcases` → 详情（单条用例详情）/ 代码（单条用例代码详情）
- **全局用例自动化代码列表**：`/testcodes` → 详情 → 跳转至对应接口的测试代码列表页
- **系统配置**：`/settings`
- **登录/注册**：`/login`、`/register`

侧栏菜单顺序建议：工作台、项目列表、接口列表、用例列表、用例自动化代码列表、系统配置；底部「退出登录」。

## 八、技术约束与选型

- **后端**：Java 17，Spring Boot 3.x，Spring Security，Spring Validation，MyBatis-Plus，MySQL，HikariCP。
- **前端**：Thymeleaf 服务端渲染，原生 JavaScript 调用 REST API，自定义 CSS（layout.css）；不使用大型前端框架。
- **AI**：openai-java SDK，对接通义千问兼容接口；Endpoint/Key/Model 可配置。
- **文档解析**：Apache POI 解析 .doc/.docx；.txt、.md 按文本处理。
- **生成测试框架**：JUnit 5（生成的代码风格）。
- **部署**：可执行 Jar，java -jar 或 mvn spring-boot:run 启动。

## 九、业务规则汇总

1. **软删除**：项目（deleted）、测试用例（deleted_at）、测试代码（deleted_at）；列表与详情只查未删除数据。
2. **AI 生成前先清理**：同一接口再次「AI生成用例」前，先软删该接口下原有用例再插入；同一用例再次「AI生成测试用例自动化代码」前，先软删该用例原有代码再插入。
3. **一用例一代码**：每条测试用例对应一条 t_test_code 记录（test_case_id 关联）。
4. **文档两步走**：先保存文档（触发标准化），再在文档列表点击「AI提取接口信息」生成接口。
5. **配置优先级**：系统配置表 > application.yml 默认值。
6. **列表统一**：每页 10 条，按 created_at 倒序，日期格式 yyyy-MM-dd HH:mm:ss。
7. **从全局列表进入详情**：接口列表「详情」→ 该接口的用例列表页；用例列表「代码」→ 该用例的代码详情页；用例自动化代码列表「详情」→ 对应接口的测试代码列表页。
